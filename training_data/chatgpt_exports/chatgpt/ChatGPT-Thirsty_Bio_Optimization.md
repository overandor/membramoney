The catch is that **Mintlify is not the best fit for direct page-posting** because its normal production flow is **Git-based**: you connect a repo with the Mintlify GitHub app, push docs files, and Mintlify deploys automatically from the default branch. Their starter kit and publishing docs both point to that repo-driven workflow, not a simple “POST markdown and publish” API. ([GitHub](https://github.com/mintlify/mintlify-docs), [Mintlify](https://mintlify.com/mintlify/docs/deploy/deployments))

So my recommendation is:

**Best for direct API posting:** **ReadMe**\
**Best for article/KB automation:** **Document360**\
**Best if you insist on Mintlify:** use a **Git-commit publisher** instead of a content API. ([ReadMe](https://docs.readme.com/main/reference/createdoc?utm_source=chatgpt.com), [ReadMe](https://docs.readme.com/main/reference/createcategory?utm_source=chatgpt.com), [apidocs.document360.com](https://apidocs.document360.com/apidocs/add-an-article?utm_source=chatgpt.com), [GitHub](https://github.com/mintlify/mintlify-docs))

Here’s the cleanest setup:

```text
LLM → your backend → normalize markdown/openapi → publish adapter
```

```python
      <div class="viewer">
```

## Minimal private prototype

```python
from fastapi import FastAPI
from pydantic import BaseModel
import queue, threading, time

app = FastAPI()
jobs = queue.PriorityQueue()
state = {}

class Job(BaseModel):
    priority: int
    slug: str
    version: int
    content: str

def worker():
    while True:
        prio, item = jobs.get()
        slug = item["slug"]
        current = state.get(slug, {}).get("version", -1)
        if item["version"] > current:
            state[slug] = item
            # push to GitHub here
        jobs.task_done()

threading.Thread(target=worker, daemon=True).start()

@app.post("/sync-doc")
def sync_doc(job: Job):
    jobs.put((-job.priority, job.dict()))
    return {"queued": True, "slug": job.slug, "version": job.version}
```
